## 一、项目背景：为什么要做这个？

### 1.1 你离开项目前的情况

云舒，你三个月前离开这个科研项目时，团队正在研究**如何让AI系统理解学术论文中的复杂约束条件**。这个问题听起来很抽象，我用一个实际例子来说明：

**典型的Browsecomp题目**:
> "找一篇2010年前发表的论文，有5位作者，其中一位作者的名字和某个国家同名，且有3位作者来自同一所大学。"

这个问题的难点在于：
- 约束条件**多而复杂**（时间、数量、名字、机构关系）
- 需要在**知识图谱中跳转**（论文→作者→机构）
- AI需要**自动生成查询推理链**

### 1.2 问题的核心

杨逸飞同学负责主项目，他面临的核心问题是：

> **"看到一个约束描述（比如'published before 2010'），应该在知识图谱中做什么操作？"**

这就像给AI提供一本"操作手册"：
- 看到"published before X" → 在当前Paper节点上过滤publication_year属性
- 看到"authored by N people" → 沿着HAS_AUTHOR边跳转到Author节点并计数
- 看到"affiliated with X university" → 沿着AFFILIATED_WITH边跳转到Institution节点

### 1.3 这个项目的作用

这个子项目就是**给AI提供这本"操作手册"** — 一张包含30条规则的映射表，每条规则告诉AI：
- 遇到什么关键词（trigger_keywords）
- 应该执行什么图谱操作（graph_operation）
- 怎么在知识图谱中跳转（traverse_edge / filter_current_node）

**重要**: 这是一个**Codebuddy辅助完成的项目**，因为：
1. 你离开项目三个月，对最新进展不太熟悉
2. 杨逸飞同学能力很强，已经在主项目中做了很多工作
3. 这个子项目是为了配合杨逸飞的主项目，提供一个标准化的接口

## 二、项目核心内容：到底做了什么？

### 2.1 核心交付物

**最重要的文件**: `constraint_to_graph_mapping.json`

这是一个包含**30条约束映射规则**的JSON文件，每条规则的结构如下：

```json
{
  "constraint_id": "C01",
  "constraint_type": "temporal",
  "constraint_name": "时间/发表年份约束",
  "trigger_keywords": [
    "published between",
    "before",
    "after",
    "in the 1970s"
  ],
  "graph_operation": {
    "action": "filter_current_node",
    "target_node": null,
    "edge_type": null,
    "filter_attribute": "publication_year",
    "filter_condition": "temporal_range"
  },
  "examples": [
    "paper published between 2015 and 2020",
    "published before 2010"
  ]
}
```

### 2.2 三种基础操作

整个系统只有**3种基础操作**（就像乐高积木只有几种基本块）：

#### 操作1: filter_current_node（在当前节点上过滤）

```json
{
  "action": "filter_current_node",
  "target_node": null,
  "edge_type": null,
  "filter_attribute": "publication_year"
}
```

**什么时候用**: 不需要跳转，只需要在当前节点上检查属性
**例子**: 
- "published before 2010" → 检查Paper的publication_year
- "born between 1933 and 1935" → 检查Author的birth_year
- "title ends with 'problems'" → 检查Paper的title

#### 操作2: traverse_edge（沿边跳转）

```json
{
  "action": "traverse_edge",
  "target_node": "Author",
  "edge_type": "HAS_AUTHOR",
  "direction": "outgoing"
}
```

**什么时候用**: 需要跳转到另一个节点类型
**例子**:
- "authored by" → Paper沿HAS_AUTHOR边跳转到Author
- "affiliated with" → Author沿AFFILIATED_WITH边跳转到Institution
- "published in" → Paper沿PUBLISHED_IN边跳转到Venue

#### 操作3: traverse_and_count（跳转并计数）

```json
{
  "action": "traverse_and_count",
  "target_node": "Author",
  "edge_type": "HAS_AUTHOR",
  "filter_condition": "count_equals"
}
```

**什么时候用**: 需要统计数量（"5位作者"、"27个引用"）
**例子**:
- "authored by five individuals" → 统计HAS_AUTHOR边连接的Author数量

### 2.3 知识图谱的结构

整个映射表严格遵循**5种节点、5种边**的图谱结构：

**节点类型** (5种):
```
┌─────────────┐
│   Paper     │ 论文
│   Author    │ 作者
│ Institution │ 机构（大学、公司）
│   Venue     │ 期刊/会议
│   Entity    │ 通用实体（奖项、主题、方法、数据等）
└─────────────┘
```

**边类型** (5种):
```
Paper ─[HAS_AUTHOR]────→ Author
Author ─[AFFILIATED_WITH]─→ Institution
Paper ─[PUBLISHED_IN]──→ Venue
Paper ─[MENTIONS]──────→ Entity
Paper ─[CITES]─────────→ Paper
```

**设计理念**: 
- ✅ **没有虚拟节点**（不会出现`EducationNode`、`AwardNode`这种额外节点）
- ✅ **严丝合缝的乐高接口**（只使用这5种节点和5种边来表达所有关系）
- ✅ **通过属性区分细节**（比如AFFILIATED_WITH边的relationship_type属性区分"就读"和"任职"）

## 三、如何理解这个设计？

### 3.1 设计思路：关键词驱动

整个系统的核心逻辑非常简单：

```python
# 伪代码
def 看到约束描述_应该做什么(constraint_text):
    for rule in 映射表的30条规则:
        if 约束文本中包含rule的关键词:
            return rule的图操作
    return "找不到匹配规则"
```

**举例**:
```python
constraint = "paper published before 2010"
# ↓
# 系统在映射表中找到C01规则
# C01的关键词包含 "before"
# ↓
# 返回操作: filter_current_node, attribute=publication_year
```

### 3.2 为什么这样设计？

#### 原因1: 杨逸飞需要"即插即用"的接口

杨逸飞的主项目是一个复杂的推理引擎，他需要一个简单明确的接口：

```python
# 杨逸飞的代码中只需要：
mapper = ConstraintMapper('constraint_to_graph_mapping.json')
operation = mapper.lookup_rule("published before 2010")

# 拿到operation后，他知道该做什么了
if operation['action'] == 'filter_current_node':
    # 在当前节点上过滤
    filter_by_attribute(operation['filter_attribute'])
elif operation['action'] == 'traverse_edge':
    # 跳转到下一个节点
    current_node = traverse(operation['edge_type'], operation['target_node'])
```

#### 原因2: 可扩展性

如果将来遇到新的约束类型，只需要在JSON文件中添加一条新规则，**不需要修改代码**：

```json
{
  "constraint_id": "C31",
  "constraint_type": "new_constraint",
  "trigger_keywords": ["new keyword"],
  "graph_operation": {...}
}
```

#### 原因3: 人类可读性

JSON格式让规则一目了然，团队其他成员可以轻松理解和修改。

### 3.3 设计亮点

1. **组合性**: 3种基础操作可以组合成任意复杂的推理链
   ```
   Paper (filter: year<2010) 
   → [HAS_AUTHOR, count=5] → Author 
   → [AFFILIATED_WITH] → Institution (filter: same)
   ```

2. **完全对齐KG Schema**: 100%使用现有的5种节点和5种边，没有引入任何额外概念

3. **关键词丰富**: 每条规则包含多个同义表达
   ```
   "affiliated with", "worked at", "faculty member at", "employed by"
   ```

- C24: 审稿角色（editor, reviewer）
- C28: 人名匹配（name identical to）

**Institution节点相关** (3条):
- C06: 成立时间（founded between）
- C21: 地理位置（located in）

## 六、如何使用这个项目？

### 6.1 最简单的使用方式（5分钟入门）

创建一个Python文件 `test_mapper.py`:

```python
import json

# 1. 加载映射表
with open('constraint_to_graph_mapping.json', 'r', encoding='utf-8') as f:
    mapping = json.load(f)

# 2. 定义查找函数
def lookup_rule(constraint_text):
    """给定约束描述，返回对应的图操作"""
    text_lower = constraint_text.lower()
    
    for rule in mapping['constraint_mappings']:
        # 检查是否有关键词匹配
        for keyword in rule['trigger_keywords']:
            if keyword.lower() in text_lower:
                return rule
    
    return None  # 找不到匹配规则

# 3. 测试
test_constraints = [
    "published before 2010",
    "authored by five individuals",
    "affiliated with Stanford University",
    "Ph.D. from MIT"
]

for constraint in test_constraints:
    rule = lookup_rule(constraint)
    if rule:
        operation = rule['graph_operation']
        print(f"\n约束: {constraint}")
        print(f"规则ID: {rule['constraint_id']}")
        print(f"操作: {operation['action']}")
        if operation['target_node']:
            print(f"目标节点: {operation['target_node']}")
        if operation['edge_type']:
            print(f"边类型: {operation['edge_type']}")
    else:
        print(f"\n约束: {constraint}")
        print("❌ 找不到匹配规则")
```

## 七、与杨逸飞项目的集成

### 7.2 接口设计

这个映射表项目为杨逸飞提供了**标准化的接口**：

```python
# 杨逸飞的代码中：
from constraint_mapper import ConstraintMapper

class BrowsecompSolver:
    def __init__(self):
        self.mapper = ConstraintMapper('constraint_to_graph_mapping.json')
        self.kg = AcademicKnowledgeGraph.load()
    
    def solve(self, question):
        # 1. 提取约束
        constraints = self.extract_constraints(question)
        
        # 2. 对每个约束查找规则
        operations = []
        for constraint in constraints:
            rule = self.mapper.lookup_rule(constraint)
            operations.append(rule['graph_operation'])
        
        # 3. 执行图谱查询
        results = self.execute_on_kg(operations)
        
        return results
```

## 八、项目的局限性与改进方向

### 8.1 已知问题

1. **关键词匹配过于简单**
   - 当前: 简单的字符串包含匹配
   - 问题: 可能有多个规则匹配同一个约束
   - 改进方向: 引入优先级机制或语义匹配

2. **测试覆盖率**
   - 当前: 30个测试用例，22个通过（73%）
   - 问题: 8个失败是因为关键词冲突
   - 改进方向: 优化关键词列表

3. **没有处理复杂组合逻辑**
   - 当前: 每条规则只处理单跳逻辑
   - 问题: "三位作者来自同一机构"这种需要聚合的逻辑需要引擎层实现
   - 说明: 这是**设计选择**，不是缺陷

### 8.2 可以改进的地方

如果你想为项目做贡献，可以考虑：

#### 改进1: 添加规则优先级

```json
{
  "constraint_id": "C01",
  "priority": 10,  // 新增：数字越大优先级越高
  "trigger_keywords": [...]
}
```

#### 改进2: 支持正则表达式匹配

```json
{
  "constraint_id": "C02",
  "trigger_patterns": [
    "authored by (\\w+) individuals",  // 可以提取"five"
    "co-authored by (\\d+) people"     // 可以提取"5"
  ]
}
```

#### 改进3: 添加示例驱动的测试

创建更多真实的Browsecomp题目作为测试用例。

#### 改进4: 性能优化

构建反向索引，将查找时间从O(n×m)优化到O(k)。

### A1. 最简单的查找函数

```python
import json

def load_mapping(file_path='constraint_to_graph_mapping.json'):
    with open(file_path, 'r', encoding='utf-8') as f:
        return json.load(f)

def lookup_rule(constraint_text, mapping):
    text_lower = constraint_text.lower()
    for rule in mapping['constraint_mappings']:
        for keyword in rule['trigger_keywords']:
            if keyword.lower() in text_lower:
                return {
                    'rule_id': rule['constraint_id'],
                    'constraint_type': rule['constraint_type'],
                    'operation': rule['graph_operation']
                }
    return None

# 使用
mapping = load_mapping()
result = lookup_rule("published before 2010", mapping)
print(result)
```

### A2. 推理链生成器

```python
class ReasoningChainGenerator:
    def __init__(self, mapping_file='constraint_to_graph_mapping.json'):
        self.mapping = load_mapping(mapping_file)
    
    def generate_chain(self, constraints, start_node="Paper"):
        current_node = start_node
        chain = []
        
        for idx, constraint in enumerate(constraints):
            result = lookup_rule(constraint, self.mapping)
            
            if result is None:
                chain.append({
                    'step': idx + 1,
                    'constraint': constraint,
                    'error': 'NO_RULE_FOUND'
                })
                continue
            
            operation = result['operation']
            action = operation['action']
            
            if action == 'filter_current_node':
                chain.append({
                    'step': idx + 1,
                    'action': 'FILTER',
                    'node': current_node,
                    'attribute': operation['filter_attribute'],
                    'constraint': constraint
                })
            
            elif action == 'traverse_edge':
                chain.append({
                    'step': idx + 1,
                    'action': 'TRAVERSE',
                    'from': current_node,
                    'edge': operation['edge_type'],
                    'to': operation['target_node'],
                    'constraint': constraint
                })
                current_node = operation['target_node']
            
            elif action == 'traverse_and_count':
                chain.append({
                    'step': idx + 1,
                    'action': 'COUNT',
                    'node': current_node,
                    'edge': operation['edge_type'],
                    'target': operation['target_node'],
                    'constraint': constraint
                })
        
        return chain

# 使用
generator = ReasoningChainGenerator()
chain = generator.generate_chain([
    "published before 2010",
    "authored by five individuals",
    "affiliated with Stanford"
])

for step in chain:
    print(f"步骤{step['step']}: {step['action']} - {step.get('constraint')}")
```

### A3. 验证脚本

```python
import json

def validate_mapping_file(file_path='constraint_to_graph_mapping.json'):
    """验证映射文件的基本结构"""
    try:
        with open(file_path, 'r', encoding='utf-8') as f:
            data = json.load(f)
        
        # 检查必需字段
        assert 'constraint_mappings' in data
        assert 'node_types' in data
        assert 'edge_types' in data
        
        # 检查规则数量
        num_rules = len(data['constraint_mappings'])
        print(f"✓ 加载了 {num_rules} 条规则")
        
        # 检查每条规则
        for rule in data['constraint_mappings']:
            assert 'constraint_id' in rule
            assert 'trigger_keywords' in rule
            assert 'graph_operation' in rule
            assert len(rule['trigger_keywords']) > 0
        
        print("✓ 所有规则格式正确")
        return True
    
    except Exception as e:
        print(f"✗ 验证失败: {e}")
        return False

# 使用
validate_mapping_file()
```
